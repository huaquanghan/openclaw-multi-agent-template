# Knowledge Routing

How to decide where information should live in the **Quả Quả + Teams** template.

## Three layers

### 1. Workspace
Use the repository workspace for:
- active execution
- drafts
- temporary artifacts
- task-specific scratch work
- local operating rules

If it changes often or only matters to the current task, it probably belongs here.

### 2. Memory files
Use memory files for:
- continuity across sessions
- user preferences
- personal context
- stable habits and working patterns
- short summaries of what happened

Typical files:
- `memory/YYYY-MM-DD.md`
- `MEMORY.md`

### 3. Durable knowledge layer
Use a separate durable knowledge system, such as GBrain, when information is:
- reusable over time
- valuable across projects or sessions
- worth searching independently of chat history
- more like knowledge than workflow residue

Examples:
- stable project summaries
- durable people or company notes
- long-lived concepts
- source-backed reference material

## Promotion rule

Do not promote too early.
A good rule is:
- workspace first when the information is still moving
- memory when it matters for continuity
- durable knowledge only when it becomes stable and reusable

## Routing examples

### Example 1: Task checklist
Store in workspace or daily memory, not durable knowledge.

### Example 2: User preference
Store in memory.

### Example 3: Reusable project operating model
Store in durable knowledge if you maintain a separate knowledge layer.

### Example 4: Research dump
Keep raw notes in workspace first. Distill only the lasting parts later.

## What not to do

Avoid:
- dumping the whole workspace into durable knowledge
- treating memory files like a scratchpad for every command output
- storing unstable drafts as if they were final knowledge

## Bottom line

Good routing keeps the system clean:
- workspace for doing
- memory for continuity
- durable knowledge for lasting meaning
